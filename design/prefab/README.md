# Aperture — Prefab Portal Design (working)

**Status:** 🟠 Working design — an active, growing record of the design session that walks the
**concrete brain-bank portal** UX first, then factors it into the configurable engine. Not yet
ADRs; decisions that crystallize here graduate to `../decisions/`. Started 2026-06-16.

## Why "prefab first"

We design the concrete, opinionated brain-bank portal — hand-built by a component designer —
**before** the generic config engine. This gives the engine a definition of done: *the config
model is correct when it can reproduce this prefab*. It matches the handoff's own framing
("the brain bank portal is the first and most-tested instance of the generic app") and the
§9.2 instinct to calibrate the abstraction against concrete targets. Concrete domain nouns
(Subject, Sample, brain_region, genotype) belong freely in these stories — they live in schema
+ config; keeping them out of *source* (ADR-0002) is the Tier-2 job.

## Grounding schema (brain-bank instance)

Source: `hippo/examples/omics/schema.yaml`. The portal is built against Hippo's autogenerated
GraphQL transport rendered from this schema.

| Entity | Key slots (annotations) |
|---|---|
| **Subject** (donor) | `external_id` (str, indexed) · `species` (enum: Homo sapiens/Mus musculus/Rattus norvegicus, req) · `biological_sex` (enum: male/female/unknown) · `age_at_collection` (float) · `diagnosis` (str, indexed, **fts**) |
| **Sample** | `external_id` (indexed) · `tissue_type` (req, indexed) · `tissue_region` (indexed, **fts**) · `collection_date` (date) · `passage` (int) |
| **BrainSample** *(is-a Sample)* | + `brain_region` (indexed, **fts**) · `hemisphere` (enum) · `post_mortem_interval_hours` (float) |
| **Datafile** | `uri` · `file_type` (enum) · `modality` (enum: RNASeq/WGBS/WGS/ATAC/genotyping/…) · `genome_build` · `is_primary` (bool) |
| **Dataset / Workflow / WorkflowRun** | named/versioned collections, pipelines, executions |
| **Relationships** | `donated` (Subject 1→N Sample) · `derived_from` (Sample N↔N) · `generated` (Sample 1→N Datafile) · `input_to`/`output_of` (Datafile N↔N WorkflowRun) · `instance_of` (WorkflowRun N→1 Workflow) · `contains` (Dataset N↔N Datafile) |

### The annotation → UI mapping (the conceptual key)

The schema already encodes most **Level-2 binding** decisions. This is *why* config-driven UI
is tractable — the baseline is derived, not authored:

| LinkML annotation | UI affordance |
|---|---|
| `indexed: true` | filterable / sortable candidate |
| `search: fts` | full-text search slot |
| `type: enum` + `values:` | facet as checklist **with counts** |
| `type: float` / `int` | numeric **range** facet + sort |
| `type: date` / `datetime` | date-range facet + sort |
| `required: true` | non-nullable (column always present) |
| relationship | navigable link; count column; relationship facet |

## Recurring implementation theme — capability negotiation (architecture Layer D)

The portal's features are **gated by what the GraphQL source supports** (architecture Layer D:
"capability-negotiated source adapter"). Hippo's v0.1 GraphQL is deliberately thin — it
advertises **equality filters + AND/OR, offset pagination, and FTS on `search:fts` slots**, but
**not** sort (`order_by`), range filters, or aggregation/counts. So every prefab feature lands
in one of three buckets:

- **(a) Supported** → use directly.
- **(b) Hippo enhancement** → propose a server capability (e.g. `order_by`, `totalCount`,
  faceted aggregation). Track as Hippo issues.
- **(c) Adapter compensation** → compute client-side in the Layer-D adapter (e.g. DuckDB-WASM
  for in-browser aggregation, ADR-0011), hidden behind the capability protocol — never exposed
  as user scripting (preserves ADR-0004).

We tag each open question (a)/(b)/(c) as we go.
